# quantix-rust 全库审计发现与优化对策（2026-03-29）

> 目标：沉淀“可执行、可追溯、可并存”的改进方案。  
> 约束：兼容 **Cursor + WSL** 并行开发，不强制单一运行方式。

---

## 1. 审计范围

本次审计聚焦三个维度：

1. 构建与测试稳定性（脚本、CI、基础入口）
2. 架构与文档一致性（README / docs/FUNCTION_MAP.md / CLI 行为）
3. 风险与维护成本（外部依赖、预期告警、环境耦合）

---

## 2. 主要发现

## 2.1 构建与环境发现

- 本地构建验证依赖 Rust 工具链；若环境未安装 `cargo`，则无法直接执行 `cargo check`。
- 本次复核环境中 `cargo` 可用，因此更准确的结论是：仓库可读性较高，但“本地可执行性”受工具链前置条件影响，而不是仓库本身缺少入口。

### 对策

- 在文档中明确区分：
  - **代码审计模式（无 Rust 工具链）**：允许仅做结构/一致性审查
  - **构建验证模式（有 Rust 工具链）**：执行 `cargo check/test/clippy`
- 增加“最小开发前置检查”说明（见第 5 节建议）。

---

## 2.2 文档一致性发现

### 发现 A：README 中存在重复段落（Phase 29 / 29B 重复）

- `README.md` 中 `Phase 29` 与 `Phase 29B` 描述出现重复块，增加维护负担，易产生未来漂移。

### 对策

- 合并重复段落，保留一处“单一事实源”。
- 在该段落末尾加“最后更新日期 + 验收脚本引用”。

### 发现 B：阶段口径存在潜在冲突

- README 中仍有“全部 20 个阶段已完成”的描述；
- 同时文档内又存在 29/29B/29C 等后续阶段内容。

### 对策

- 统一口径为：
  - “历史基础阶段 1-20 已完成”；
  - “后续执行主线阶段（29/29B/29C 等）按里程碑持续演进”。
- 避免“全部完成”这类绝对表述，改为“截至日期的完成面”。

### 发现 C：Bridge 能力描述需区分“API 能力”与“产品承诺”

> 历史说明：
> 本节记录的是 2026-03-29 审计时点的判断。
> 当前项目事实已变化：仓库现已提供受能力门控的 `qmt_live` 真实提交通道；
> 通用 `live` 语义仍未实现，但“当前对外承诺仍以 preview-only 为准”已不再是现状描述。

- `docs/FUNCTION_MAP.md` 提到 `/api/v1/broker/qmt/orders/live` 与“QMT 真实下单”树节点；
- README 侧强调 `QMT preview-only`。

### 对策

- 在 `docs/FUNCTION_MAP.md` 对应位置增加注记：
  - `orders/live` 为桥接 API 端点能力描述；
  - 当时的 quantix-rust 对外承诺仍以 `preview-only` 为准。
- 避免用户误解“端点存在 = 产品已开放”。

---

## 2.3 验收脚本与实际行为发现

- `scripts/verify_features.sh` 对“预期未完成功能”采用 `run_expect_warn` 模式，策略合理。
- 当前 `Parquet export placeholder` 的 hint 与现实现状一致（命中“暂未实现/未实现/unsupported”）；若后续实现改为“未找到数据/empty/no data”等返回文案，现有 hint 可能出现误判。

### 对策

- 前瞻性扩展 hint 关键词覆盖面（如“未找到数据/empty/no data”），减少后续演进时的误判。
- 保留“WARN 是预期”机制，不将其误当失败。

---

## 2.4 CI 维护性发现

- `.github/workflows/ci.yml` 内容完整，但偏重型：
  - 包含 coverage、outdated、多服务依赖，执行成本高；
  - 对 fork/低配环境不够友好。

### 对策

- 增加分层执行策略：
  - PR 默认：lint + unit tests（快）
  - main/nightly：coverage + benchmark + outdated（重）
- 提升反馈速度与稳定性，降低“非功能性失败”占比。

---

## 3. 风险清单（按优先级）

### P0（高优先）

1. README 重复与口径冲突（易误导新贡献者）
2. Bridge 描述不完全对齐（易引发能力误判）

### P1（中优先）

1. 验收脚本 hint 对未来返回文案覆盖不足，存在后续误判风险
2. CI 任务未分层导致贡献门槛偏高

### P2（持续优化）

1. 为各模块建立统一“文档事实源”索引（减少多处复制）
2. 引入定期文档一致性审查（比如每周/每里程碑）

## 4. Cursor + WSL 并存开发策略（重点）

原则：**允许并存，不互斥，不强制迁移。**

## 4.1 运行形态建议

- **WSL 主执行**：Rust 构建、脚本运行、路径基于 Linux。
- **Cursor 主编辑**：跨文件修改、审阅、重构、文档维护。
- 二者并存，不要求“只在一个环境完成全部流程”。

## 4.2 路径与配置约束

- 文档示例统一优先使用 Linux 路径（WSL 一致性最好）。
- Windows Bridge 路径继续保留：`/mnt/d/mystocks/quantix/quantix_bridge`。
- 对可覆写路径一律提供环境变量入口（已有约定保持不变）。

## 4.3 并存原则（建议写入团队约定）

1. 不依赖 Cursor 专有能力作为唯一执行路径。
2. 不依赖 Windows 专有路径作为唯一数据路径。
3. 所有关键流程至少保留一条纯 CLI 可执行路径。
4. 文档中所有“必须”项应可在 WSL 独立完成。

## 6. 结论

- 当前仓库在功能覆盖与命令体系上已具备较强完整度。
- 主要优化空间在“**文档一致性、能力边界表达、验证策略分层**”。
- 采用“Cursor + WSL 并存”策略可在不破坏现有流程的前提下提升协作效率与可维护性。
