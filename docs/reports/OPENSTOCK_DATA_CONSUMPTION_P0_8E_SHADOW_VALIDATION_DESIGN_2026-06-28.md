# OpenStock Data Consumption P0.8e Shadow Validation Design

Date: 2026-06-28

## Decision

P0.8e is split into two stages:

1. Shadow validation design gate.
2. Future opt-in persistence implementation.

This slice completes only stage 1. It does not approve or implement any ClickHouse write path.

## Scope

P0.8e shadow validation should prove that OpenStock daily kline data can be compared against the project's canonical kline model and a future ClickHouse shadow target without changing runtime data-source behavior.

Allowed now:

- Document the shadow validation contract.
- Define schema mapping, deduplication keys, rollback requirements, and dry-run gates.
- Record GitNexus impact targets for any future implementation.

Not allowed now:

- No production Rust source changes.
- No live OpenStock network calls.
- No ClickHouse writes.
- No data-source route replacement.
- No qmt_live or miniQMT changes.
- No `ExecutionAdapter` or `OrderStatus` changes.
- No `.unwrap()` cleanup.

## Input Contract

The first implementation candidate must use the already committed local fixture path:

- Source artifact: `tests/fixtures/openstock/daily_kline.json`
- Parser: `parse_daily_kline_json`
- Canonical output: `Vec<Kline>`
- Validation source label: `openstock_shadow_dry_run`

Live OpenStock endpoints remain out of scope until a separate provider contract slice approves configuration, timeout, retry, rate-limit, and fixture-vs-live parity gates.

## Canonical Schema Mapping

The shadow validation model should map OpenStock daily kline rows to existing `Kline` semantics before any storage comparison.

| Shadow field | Canonical source | Required | Notes |
| --- | --- | --- | --- |
| `source` | constant | yes | Must be `openstock_shadow_dry_run` for this stage. |
| `period` | payload period | yes | Only `daily` is accepted in the current parser. |
| `code` | `Kline.code` | yes | Mixed-code payloads are rejected before validation. |
| `date` | `Kline.date` | yes | Existing parser requires `%Y-%m-%d`. |
| `open` | `Kline.open` | yes | Decimal-preserving comparison. |
| `high` | `Kline.high` | yes | Parser rejects `high < low`. |
| `low` | `Kline.low` | yes | Decimal-preserving comparison. |
| `close` | `Kline.close` | yes | Already used by the P0.8d analysis fixture loop. |
| `volume` | `Kline.volume` | yes | Integer only. |
| `amount` | `Kline.amount` | no | Optional until live provider evidence proves field availability. |
| `adjust_type` | `Kline.adjust_type` | yes | Existing parser accepts `none`, `qfq`, and `hfq`. |
| `batch_id` | generated by validator | yes | Required for rollback if persistence is later approved. |
| `artifact_hash` | fixture file hash | yes | Required to make dry-run evidence reproducible. |

## Deduplication Contract

The dry-run report must compute a deterministic identity key before any future persistence is considered:

```text
source + period + code + date + adjust_type
```

For fixture-backed validation, `source` must remain `openstock_shadow_dry_run`.

Any future live provider must use a different source label, for example `openstock_live_shadow`, so fixture and live evidence cannot collide.

The first implementation should fail closed if duplicate keys are found inside one artifact.

## Dry-Run Gates

All P0.8e implementations must remain dry-run by default.

A valid dry-run report must include:

- `writes_performed=false`
- `target=shadow_validation_report`
- artifact path
- artifact hash
- row count
- code set
- date range
- adjust type set
- duplicate key count
- rejected row count
- parser error, when applicable

The CLI or service name is intentionally not assigned in this design slice. That decision belongs to the implementation slice after a fresh GitNexus impact check.

## Rollback Requirements Before Any Write Path

No ClickHouse write path may be implemented until a later approved slice defines all of the following:

- Shadow table name and database namespace.
- Full shadow table schema.
- Batch identity column.
- Source artifact hash column.
- Insert mode.
- Delete-by-batch rollback command.
- Operator runbook.
- Dry-run preview output.
- Failure behavior for partial writes.
- CI coverage proving normal runs do not write ClickHouse.

The minimum rollback identity is:

```text
batch_id + source + artifact_hash
```

The `ControlledPersistencePolicy` result is the main architectural boundary: OpenStock may borrow dry-run discipline from prior miniQMT work, but it must not couple to miniQMT policy types or storage semantics.

## Implementation Plan For The Next Slice

The next safe slice is P0.8e-2: test-only shadow validation report builder.

Suggested constraints:

- Add no ClickHouse write code.
- Use only the committed fixture.
- Produce an in-memory report struct or test helper.
- Assert `writes_performed=false`.
- Assert row count, code set, date range, artifact hash, and duplicate key count.
- Keep execution adapters and qmt_live untouched.
